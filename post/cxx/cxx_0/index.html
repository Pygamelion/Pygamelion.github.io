C++ 的许多设计决策根源于我对强迫人按某种特定方式行事的极度厌恶。历史上一些最坏的灾难就起因于理想主义者们试图强迫人们“做某些对他们最好的事情”。这种理想主义不仅导致了对无辜者的伤害,也迷惑和腐化了施展权利的理想主义者自身。我还发现,当经验和实验与其教义或理论出现不寻常的冲突时,理想主义者往往有忽略它们的倾向。在理想出问题的地方,甚至权威人士也准备赞成的时候,我则趋向于提供一些支持,使程序员有自己选择的权力…

上一次学Cxx还是刚上大学，随大流想考一个计算机二级证书，觉得office主要是文字处理没有意思，这门语言早在中学时代就刻入把名字刻在我脑袋里了，毕竟当时手机上玩我的世界首页都会飘着一行“Build with C++”。清澈愚蠢的我随大流整了本大名鼎鼎的谭浩强《C++程序设计》, 学了两章就被诸如i*(i+=--i)、a=(a++)+(++a)的窒息语法吓到，随后开始对各种写法排列组合。虽然最后还是拿到证书，我对整个计算机技术的学习也到此为止了(毕竟考试的其实很简单，真没有什么动力再玩这些花式trick了(当时稍微知道计算机学生们天天在用电路做加法乘法二进制呢，怎么能把大学当成小学来读！来自高考数学上不了100分的老子的鄙视

之后莫名走入物理学的花花世界，开开心心地学了一段时间，各种真假难辨的物理理论充斥我的脑袋，让我深感自己只是物理学的观光者，没有坚定信念和探险精神 practical的方向与academic资源，期间Python、Julia、Rust等编程语言都有接触，虽然身为LHAASO组成员，我一直想绕着C++和Root走，足见当年心理阴影之深。今天回想起来，已经是时过境迁，兜兜转转回到了True or False的比特世界 世界上又多了个普通转行物理人，却少了个高级程序员(；′⌒`) 现在回来重学Cxx，好处是AI领路、Cxx标准更新、工具链换掉了一批，网络上各种学习资源随手可得，其它语言的学习也让我重新认识了Cxx是多么强大的存在。毕竟这么多年过去，当年奇怪写法的遗毒也在我脑中消退了，不好说是好事还是坏事…

欲买桂花同载酒，终不似，少年游……

从编译工具开始

由于沉重的历史包袱，Cxx不能像其它语言一样方便地执行…对于数值计算来说Julia比C++好用多力

  • 静态库：链接后会直接把里面所有的代码都合并生成到可执行程序中
  • 动态库：链接后仅将其中导出的函数提供给可执行程序用，实际运行过程中还是会单独加载这个动态库 (长.so .dll这样的

总之已经可以尽量不要去管太多细节的东西了，除非需要展示自己有20年Cxx功力o((>ω< ))o

Visual Studio里面一套东西还没搞明白怎么配置的，先用着Vscode吧 ＞︿＜…

Cmake有些学不明白，但是有AI工具加持，对于个人学习来写小项目已经不是问题了

现在有好用的Xmake，不用是(≧∀≦)ゞ笨蛋！

Xmake通过描述域、脚本域两种不同的配置方式，隔离 80% 的简单配置以及 20% 的复杂配置，集成了Lua的脚本功能，还有更强的封装 陈皓大佬的lua教程

  • 描述域：使用 set_xx、add_xxx 等配置接口进行的配置区域
  • 脚本域：使用 on_xx、after_xxx 和 before_xxx 等配置接口的内部区域

为了优化编译速度，我们不需要编译每个源文件都去重复预处理一遍里面的头文件，可以把大部分项目常用的头文件放置在统一的头文件中预先编译，在链接阶段直接复用 Xmake 使用 set_pcxxheader("src/stdafx.h")

使用与生态兼容

终端输入xmake lua打开REPL

  • xmake lua xmake_modules/mymod.lua hello 直接绕过编译调试单个脚本
  • xmake lua lib.detect.find_tool gcc 调用模块检查编译工具链/使用模块命令
  • xmake lua find_packages zlib 查找本地安装包zlib

xmake 提供了 $(varname) 的语法支持在配置字符串中直接传递内置变量 $(env varname) 内置变量，在字符串中获取指定环境变量 xmake l os.getenvs 以打印环境变量

  • $(buildir)：实际构建输出目录
  • $(tmpdir)：临时目录
  • $(curdir)：当前运行目录
  • $(scriptdir)：当前 xmake.lua 配置脚本所在目录
  • $(programdir)：xmake 安装程序的 lua 脚本根目录

$(shell command) 内置变量的方式，在描述域配置中快速执行 shell 命令，并嵌入执行输出结果到配置字符串中

target("hello")
    set_kind("binary")
    add_files("src/*.cpp")
    if is_plat("linux") then
        add_ldflags("$(shell pkg-config --libs sqlite3)")
    end

只要是在 xmake f/config 配置命令中的所有参数选项配置，都可以通过内置变量的方式来获取使用，可以通过xmake f --help来查看配置菜单列表获取到，比较常用的有$(plat)、$(arch)、$(host)和$(mode)…

Xmake可以直接生成CMakelists.txt，但CMake还是得学，学外语好啊(

xmake project -k makefile
xmake project -k cmake
`Xmake`有Vscode插件，适合脑容量非常小，一点命令都记不住的我

下载插件后 Shift + Ctrl + p 打开一个输入命令的窗口，直接Xmake create 就能根据模板选择创建项目了

我觉得基本上来说，这些工具封装一层大佬才会用的命令行设置参数

add_rules("mode.debug", "mode.release")

target("quickapp")
    add_rules("qt.quickapp")
    add_headerfiles("src/*.h")
    add_files("src/*.cpp")
    add_files("src/qml.qrc")
    add_frameworks("QtXml") -- 添加依赖库QtXml

add_framework可以配置除QtGUI、QtCore等基础库之外的额外Qt依赖

add_links 仅仅只会简单地将库以 -lQtGui 方式来链接；而 add_frameworks主要用于添加 QT 相关库，除了会添加 -lQtGui 之外，还会添加 -I/xxx/QtGui 等头文件搜索路径，以及链接搜索路径，另外还会添加对应的 -DQT_XML_LIB 宏定义：比起手动 add_links、add_includedirs 更加的方便且跨平台
