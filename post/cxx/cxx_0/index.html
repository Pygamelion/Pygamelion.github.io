可见性和命名空间

  • 无链接 → 函数里面的local variables
  • 内部链接 → 显式使用static 关键字或者匿名namespace下的东西，当前文件私有，其它文件不可见
  • 外部链接 → 全局变量这样的东西，可以跨文件访问
  • 模块链接 → Cxx20的模块中，显式export的符号会被模块外部访问

inline 关键字现在的主要目的是允许同一个符号在多个翻译单元中出现多个定义，同时不违反 ODR（One Definition Rule）inline 负责处理ODR问题(模板编程中常用)，static 才负责内部链接，两个关键字可以一起使用static inline int wtf = 2;

如果是多个文件共享某个函数或者变量，只在一个地方定义，其它地方使用extern声明它，表示它是外部引入的但是你已经完全被inline给替代了鸭

为了防止命名冲突，可以狠狠嵌套命名空间然后起别名 命名空间同时也能起到让函数的实现和#include顺序强相关的作用

prefix::literally::everything::all::the::time();
namespace aliasprefix = prefix::literally::everything::all::the;
aliasprefix::time();

嵌套命名空间允许像创建文件目录一样对代码分层分组，Cxx 17还有对应语法糖namespace A::inline B::C {...} 等价于嵌套写法，但建议是每个项目只有一个独立的顶级命名空间，inline能在命名空间中起到符号穿透的作用

宏不属于任何命名空间，namespace对#define的宏完全没有作用，因此导致#include 头文件顺序敏感 (核心原因，此外某些库会依赖更老版本的库，所以新引入的东西一定要在下面e.g. 先#include <winsock2.h>再#include <windows.h>)

代码组织方式

头文件.h或.hpp；源文件.cpp，通常情况下，我们会使用头文件来分离声明和定义: 声明放在头文件中，而实现放在 C++ 文件中。 如果 C++ 文件中的实现发生任何更改，就不需要重新编译包含相应实现的用户代码

由于#include这样的语句只是单纯的文本复制粘贴，不参与编译语义的理解，#progma once这样的header guard只能阻止同一翻译单元内重复的代码被编译多次

#if !defined(HEADERFILE_H)
#define HEADERFILE_H
  // 在这个里面写头文件
  // 据我的clang-tidy(开启了"portability-*")提示这样写更跨平台？
#endif // HEADERFILE_H

头文件太多的话编译比较慢、#include顺序也很重要，据说还是和万恶的宏有关

Cxx标准保证除了C头文件外，所有的标准库都能够被import，但是并不是所有的第三方库都是header unit，好像取决于它们的实现方式…(以下内容摘录到这里，留待实际使用时踩坑)

模组的名字和模组文件的名字可以不一致，小小的module组成大module，以便用户使用时根据需要取用~

子module组成大module和inline namespace有点像：在大模块中使用export import 子模块可以将子模块的export内容提升到外部空间，不复制具体实现，用户能直接访问，但是在语义控制上，子module的方式更加方便管理

  • export import 子module写法只会传播export的符号，不会传播具体实现细节…这个写法header unit好像不能用
  • module 要是太大了还是建议划分接口模块.cppm/.ixx和.cpp实现(Module Implementation Unit)，实现的.cpp必须有module module_name代表这是对module的实现，实现的.cpp不处理export
  • 一个大模块也可以拆成几个部分，export module <主要module>:<划分module>这来定义一个部分，然后在主模块里拼接起来 export import:<划分module>，划分的部分不能被直接import 这才是使用的最主要原因：不给你们用小部件🤣

使用module:private，可以将声明和定义分离到单个模块文件中，如果对实现的任何更改在语义上不会影响用户代码，编译器 可能不需要 重新导入这些更改(查阅cppreference)尝试过后发现发现也许是美好的幻想？如果编译工具能够只比较接口部分、不关心实现就行？现在如果module太大还是尽可能将模块分为接口模块和实现模块

总之module至少有语义上的好处吧？ private ≈ static 提供模块的当前文件内更精细的可见性控制

  • module们可以以任何顺序导入，有更加精细的使用和控制方法，语义上的封装性更好
  • 预处理器的任何指令不会影响imported header unit, 不会像#include 头文件一样被宏影响了

## 编译模块接口并生成 IFC 和 OBJ
cl /std:c++latest /EHsc /c mathlib.cppm /interface
## 编译 main.cpp，并导入模块接口
cl /std:c++latest /EHsc /c main.cpp /module:reference mathlib.ifc
## 链接
cl /std:c++latest main.obj mathlib.obj
